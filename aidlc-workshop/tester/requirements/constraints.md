# 제약사항 (Constraints)

## 소개

본 문서는 AI 기본법 준수 확인 챗봇 서비스 개발 및 운영에 적용되는 기술적, 법적, 운영적 제약사항을 정의합니다.

---

## 1. 기술적 제약사항 (Technical Constraints)

### 1.1 기술 스택

- 백엔드는 Python으로 구현해야 한다.
- 프론트엔드는 React로 구현해야 한다.
- 백엔드와 프론트엔드는 REST API 또는 WebSocket을 통해 통신해야 한다.

### 1.2 성능

- 챗봇의 평균 응답 시간은 30초를 초과해서는 안 된다.
- 시스템은 동시 접속 사용자 100명을 처리할 수 있어야 한다.
- 프론트엔드 초기 로딩 시간은 3초 이내여야 한다.

### 1.3 LLM 연동

- LLM 백엔드로 OpenAI의 GPT-4o mini 모델을 사용해야 한다.
- GPT-4o mini API 호출 실패 시 재시도 로직(최대 3회)을 구현해야 한다.
- GPT-4o mini 응답 타임아웃은 60초로 설정해야 한다.
- GPT-4o mini 프롬프트에는 AI 기본법 관련 컨텍스트를 포함해야 한다.
- OpenAI API 키는 환경 변수로 관리하며 코드에 하드코딩해서는 안 된다.

### 1.4 데이터베이스

- AI 기본법 조항 데이터는 로컬 SQLite DB에 저장하여 관리해야 한다.
- SQLite DB 스키마는 조(Article) → 항(Paragraph) → 호(Subparagraph) → 목(Item) 4단계 계층 구조를 지원해야 한다.
- 각 계층은 고유 ID, 상위 계층 참조(외래 키), 번호, 내용, 카테고리, 시행일 필드를 포함해야 한다.
- 조항 간 계층 관계는 외래 키(Foreign Key)를 통해 표현해야 한다.
- AI 기본법 원문 텍스트는 GPT-4o mini API를 활용하여 구조화된 JSON으로 자동 파싱한 후 DB에 저장할 수 있다.
- 각 조항(article)의 임베딩 벡터는 별도 `embeddings` 테이블에 BLOB으로 저장해야 한다.
- AI 기본법 개정 시 SQLite DB를 업데이트할 수 있는 마이그레이션 구조를 갖춰야 한다.

### 1.8 임베딩 및 벡터 검색

- 임베딩 모델은 OpenAI `text-embedding-3-small`을 사용해야 한다.
- 임베딩 생성은 `POST /admin/embed-law` 및 `POST /admin/embed-guideline` 엔드포인트 호출 시 수행해야 한다.
- 임베딩 벡터는 numpy float32 배열로 직렬화하여 SQLite BLOB으로 저장해야 한다.
- 벡터 검색은 코사인 유사도(cosine similarity)를 사용하며, 별도 벡터 DB 없이 numpy로 구현해야 한다.
- Query Expansion은 사용자 입력을 GPT-4o mini로 처리하여 법적 쟁점 키워드 목록을 추출하는 방식으로 구현해야 한다.
- Query Expansion 결과와 원본 질문을 결합하여 임베딩 검색에 사용해야 한다.
- 임베딩이 없는 경우(DB 미초기화) LIKE 기반 키워드 검색으로 폴백(fallback)해야 한다.

### 1.9 가이드라인 처리

- 가이드라인 PDF는 `data/guidelines/` 폴더에 위치해야 한다.
- 텍스트 추출 가능한 PDF는 pdfplumber로, 스캔본 PDF는 GPT-4o Vision OCR로 처리해야 한다.
- 가이드라인 청크는 페이지 단위로 분할하며, 100자 미만 청크는 스킵해야 한다.
- 각 가이드라인 청크는 `related_articles` 필드에 관련 AI 기본법 조항 번호를 JSON 배열로 저장해야 한다.
- 태깅은 `POST /admin/tag-guidelines` 엔드포인트 호출 시 GPT-4o mini로 자동 수행해야 한다.
- RAG 검색 시 법령 조항 3개 + 태그 기반 가이드라인 청크 2개를 우선 조합하여 GPT에 전달해야 한다.

### 1.5 데이터

- 사용자 입력 데이터는 서버에 영구 저장하지 않는 것을 기본으로 한다. (세션 종료 시 삭제)

### 1.6 Rate Limiting

- IP 기반 Rate Limiting은 서버 측에서 구현해야 한다.
- 동일 IP에서 10분 이내 5회 초과 요청 시 HTTP 429 상태 코드와 함께 안내 메시지를 반환해야 한다.
- Rate Limiting 카운터는 Python 인메모리(딕셔너리 등)로 관리해야 한다.
- 서버 재시작 시 Rate Limiting 카운터가 초기화되는 것은 허용된 동작이다.
- Rate Limiting 로직은 사용자 인증 없이 IP 주소만으로 동작해야 한다.

### 1.7 법령 파싱 엔드포인트

- `POST /admin/parse-law` 엔드포인트는 환경변수 `ADMIN_SECRET_KEY` 기반 시크릿 키로 보호해야 한다.
- `POST /admin/embed-law` 엔드포인트는 동일한 `ADMIN_SECRET_KEY`로 보호해야 한다.
- 요청 헤더 `X-Admin-Key`가 없거나 `ADMIN_SECRET_KEY`와 일치하지 않으면 HTTP 401을 반환해야 한다.
- 파싱 대상 파일은 프로젝트 루트에 위치한 AI 기본법 원문 파일(텍스트 또는 PDF)로 고정한다.
- 파싱 중 오류 발생 시 DB를 부분 업데이트하지 않고 트랜잭션을 롤백해야 한다.
- 임베딩 생성 중 오류 발생 시 해당 조항의 임베딩을 건너뛰고 계속 진행하며 오류를 로그로 기록해야 한다.
- `ADMIN_SECRET_KEY`는 환경변수로 관리하며 코드에 하드코딩해서는 안 된다.

### 2.1 AI 기본법 관련

- 챗봇이 제공하는 분석 결과는 법적 조언이 아님을 명시해야 한다.
- 분석 결과에는 "본 결과는 참고용이며, 법적 효력이 없습니다. 정확한 법률 해석은 전문가에게 문의하세요."라는 면책 조항을 포함해야 한다.
- 분석에 사용되는 AI 기본법 조항 정보는 최신 공포 버전을 기준으로 해야 한다.

### 2.2 개인정보 보호

- 서비스는 개인정보 보호법(PIPA)을 준수해야 한다.
- 사용자 식별 정보를 수집하는 경우, 개인정보 처리방침을 명시하고 동의를 받아야 한다.
- 사용자 입력에 포함된 개인정보는 분석 목적 외에 사용해서는 안 된다.

### 2.3 저작권

- 서비스에 사용되는 오픈소스 라이브러리는 라이선스를 확인하고 준수해야 한다.
- AI 기본법 원문 인용 시 출처를 명시해야 한다.

## 3. 운영적 제약사항 (Operational Constraints)

### 3.1 가용성

- 서비스 목표 가용성은 99% 이상이어야 한다. (월간 기준 약 7.2시간 이하의 다운타임 허용)
- 계획된 유지보수는 사전에 사용자에게 공지해야 한다.

### 3.2 보안

- 모든 통신은 HTTPS(TLS 1.2 이상)를 사용해야 한다.
- 본 서비스는 완전 익명 서비스로, 회원가입 및 로그인 기능을 제공하지 않는다.
- OpenAI API 키는 환경 변수로 관리하며 코드에 하드코딩해서는 안 된다. (1.3항 참조)
- 사용자 입력에 대한 입력값 검증(Input Validation)을 수행해야 한다.

### 3.3 모니터링 및 로깅

- 시스템 오류 및 API 호출 실패는 로그로 기록해야 한다.
- 로그에는 사용자 입력 원문을 포함해서는 안 된다. (개인정보 보호)
- 서비스 응답 시간 및 오류율을 모니터링해야 한다.

### 3.4 유지보수

- AI 기본법 개정 시 분석 기준을 업데이트할 수 있는 구조로 설계해야 한다.
- 코드는 Python PEP 8 및 React 공식 스타일 가이드를 따라야 한다.

## 4. 범위 제약사항 (Scope Constraints)

- 본 서비스는 한국 AI 기본법만을 분석 대상으로 한다. (타국 법률 분석 제외)
- 본 서비스는 법률 전문가 수준의 법적 해석을 제공하지 않는다.
- 본 서비스는 완전 익명 서비스로, 회원가입/로그인 기능을 제공하지 않는다.
- 내보내기 형식은 클립보드 텍스트 복사만 지원한다. (PDF 등 파일 다운로드 미지원)
- 초기 버전(MVP)에서는 한국어 입력만 지원한다.

## 5. Lessons Learned

### 5.1 SSE 스트리밍과 로그 저장 분리

**문제**: FastAPI의 `StreamingResponse` 제너레이터(`async def event_stream()`) 내부에서 `yield "data: [DONE]\n\n"` 이후 코드는 실행되지 않는다. `BackgroundTasks`도 `StreamingResponse`와 함께 사용 시 동작하지 않는다.

**원인**: SSE 스트리밍 응답은 클라이언트가 연결을 닫으면 제너레이터가 중단되며, `[DONE]` 이벤트 전송 후 추가 코드 실행이 보장되지 않는다.

**해결책**: 스트리밍과 로그 저장을 완전히 분리한다.
- 백엔드: `POST /api/chat/log` 전용 엔드포인트를 별도로 구현
- 프론트엔드: SSE `done` 이벤트 수신 후 별도 API 호출로 로그 저장

**제약사항 추가**:
- 대화 로그 저장은 SSE 스트리밍 엔드포인트 내부에서 수행해서는 안 된다.
- 로그 저장은 프론트엔드가 응답 완료를 확인한 후 별도 API 호출로 처리해야 한다.

### 5.2 대화 로그 중복 저장 방지

**문제**: React의 상태 업데이트 콜백(`setState`)이 개발 환경에서 두 번 실행될 수 있어 동일 `session_id`로 로그가 중복 저장됐다.

**해결책**:
- `conversation_logs.session_id` 컬럼에 `UNIQUE` 인덱스 적용
- `INSERT ... ON CONFLICT(session_id) DO UPDATE SET messages = excluded.messages` 구문으로 중복 시 업데이트 처리
- 프론트엔드에서 `React.StrictMode` 제거 (개발 환경 이중 실행 방지)

**제약사항 추가**:
- `conversation_logs` 테이블의 `session_id` 컬럼은 UNIQUE 제약을 가져야 한다.
- 로그 저장 시 `INSERT OR REPLACE` 또는 `ON CONFLICT DO UPDATE` 방식을 사용해야 한다.
